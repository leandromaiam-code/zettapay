## 1. Zero Open Graph image / url on any page — HIGH — S

### Evidence
- `grep -l 'og:image' public/**/*.html` → **zero matches**.
- `grep -c 'og:url' public/index.html public/pricing.html public/signup.html public/checkout.html` → **0,0,0,0**.
- `og:title` and `og:description` are present on most pages, but the image preview is the load-bearing one for social shares.

### Why it matters
Every share on Twitter/X, LinkedIn, Discord, Slack, Telegram, iMessage, Notion, etc. will render as a text-only link with no thumbnail. For a developer-launch product where the funnel begins with social posts and DM forwards, this is the single highest-leverage SEO/marketing gap. It costs nothing to fix and compounds on every share for the lifetime of the brand.

### Suggested fix
1. Design a 1200×630 PNG (Forest #0a1612 background, Brass #d4a961 ZettaPay wordmark + tagline). Drop at `public/og-image.png`.
2. Add to every page's `<head>`:
   ```html
   <meta property="og:image" content="https://zettapay.vercel.app/og-image.png" />
   <meta property="og:image:width" content="1200" />
   <meta property="og:image:height" content="630" />
   <meta property="og:url" content="https://zettapay.vercel.app/<path>" />
   <meta property="og:type" content="website" />
   <meta name="twitter:card" content="summary_large_image" />
   <meta name="twitter:image" content="https://zettapay.vercel.app/og-image.png" />
   ```
3. Validate with `https://www.opengraph.xyz/` after deploy.

**Effort: S** — one image asset + a search-and-replace across 20 files.

## 2. Footer link `/simulate/demo` is broken — HIGH — S

### Evidence
- `public/index.html:613` →
  `<a href="/simulate/demo" class="hover:text-white transition">Simulate</a>`
- `vercel.json:35-36` only defines:
  ```json
  { "source": "/simulate/:merchant", "destination": "/api/simulate/:merchant" }
  ```
- The destination is a **serverless API function**, not a UI route. Even when a merchant slug matches, clicking the link dumps raw JSON in the browser — `demo` is not a registered merchant, so the request 404s on the API side.

### Why it matters
This is the literal Sherlock signal of an unfinished site: a footer CTA on the **homepage** that 404s or shows JSON. Anyone scanning the footer (investors, partners, the first 100 dev users) will tap it.

### Suggested fix
Pick one of:
- **(A) Remove the link** from `public/index.html` footer until a public-facing demo page exists.
- **(B) Ship a real demo page** at `public/simulate.html` (or under `/playground`) that auto-invokes `/api/simulate/<seed-merchant>` and renders the response as a styled timeline. This is also a great DevRel asset.
- **(C) Add a rewrite** `/simulate/demo → /simulate.html` and ship the static page.

**Effort: S** (option A) — **M** (option B).

## 3. `robots.txt` and `sitemap.xml` absent — HIGH — S

### Evidence
- `ls public/robots.txt public/sitemap.xml` → both `No such file or directory`.

### Why it matters
- Without `robots.txt`, search engine crawlers can hit `/api/*`, `/dashboard/*`, `/checkout/*`, and any preview/staging URL that leaks into a sitemap or backlink. Auth-gated pages risk being indexed.
- Without `sitemap.xml`, Google/Bing rely on link discovery alone; pages buried behind JS (e.g., `/docs/embed`, `/docs/faucet`) take weeks instead of hours to be indexed.
- Combined cost to ship: ~15 minutes. Combined upside: full crawl coverage from day one of launch.

### Suggested fix
Create `public/robots.txt`:
```
User-agent: *
Allow: /
Disallow: /api/
Disallow: /dashboard/
Disallow: /checkout/
Sitemap: https://zettapay.vercel.app/sitemap.xml
```
Create `public/sitemap.xml` with all 20 public HTML pages (one `<url>` block each, with `<lastmod>` from `git log -1 --format=%cI <file>` and a sensible `<priority>`).

**Effort: S** — both files together are < 100 LOC of plain text.

## 4. `<html lang>` inconsistency — MEDIUM — S

### Evidence
- 19 pages: `<html lang="en">`
- 1 page: `<html lang="pt-BR">` → `public/pay.html:2`

### Why it matters
Two real consequences:
1. **Accessibility:** screen readers switch pronunciation rules based on `lang`. `pay.html` will be read with Portuguese phonemes for English content (or vice-versa, depending on the actual body copy), which is broken for both PT and EN users.
2. **SEO hreflang:** Google treats `lang` as a hint for the geo/language version it serves. One outlier page gets misclassified.

### Wider implication
`CLAUDE.md` rule 27 says **"PT-BR como default de UI; i18n via dictionary keys, sem strings hardcoded"** — yet the current site ships English-only HTML with strings hardcoded throughout. This is a larger architectural call: either explicitly **drop the PT-BR-default rule for the global launch site** (and remove the `pay.html` outlier), or **build the i18n layer** before launch. The decision is Leandro's; the audit only flags the inconsistency.

### Suggested fix
- **Minimal:** change `pay.html:2` to `lang="en"` to match the rest of the site.
- **Strategic:** decide whether the launch site is English-only (most AI-agent dev audiences are global EN), then either remove the PT-BR rule from `CLAUDE.md` or schedule a real i18n mission (Z-future).

**Effort: S** (minimal) — **L** (full i18n).

## 5. Signup → dashboard handoff routes new merchants to the legacy dashboard — HIGH — M

### Evidence
- `public/signup.html:497` →
  `<a id="dashboardLink" href="/dashboard" class="btn-mint ...">View Dashboard</a>`
- `public/signup.html:808` →
  `dashboardLink.setAttribute('href', result.dashboard_url || '/dashboard');`
- `vercel.json:66-71` →
  ```json
  { "source": "/dashboard",           "destination": "/dashboard.html" },
  { "source": "/dashboard/:merchant", "destination": "/dashboard/index.html" }
  ```
- `wc -l public/dashboard.html` = **1,813** lines (the **legacy** dashboard — see memory `project_dashboard_analytics_split.md`).
- `wc -l public/dashboard/index.html` = **1,613** lines (the **new** dashboard, wallet-less since PR #164).

### Why it matters
The fallback path is the failure mode:
- The `||` fallback at `signup.html:808` fires whenever the `/merchants/register` API response is missing the `dashboard_url` field — any backend bug, error response, or older API build silently sends a brand-new merchant to the **legacy** dashboard.
- The legacy dashboard exists for backwards compat but is no longer the supported merchant surface. New merchants seeing it will get an inconsistent UX vs. what `/docs` and screenshots show, and will likely think "I just signed up, where's MY data?"
- Additionally, the nav link in `signup.html:262` is hard-coded to `/dashboard` (legacy) — anyone returning to the site and clicking "Dashboard" in the nav before knowing their slug also hits the legacy page.

### Suggested fix
1. **Verify the API always returns `dashboard_url`.** Add a server-side test asserting `POST /merchants/register` response contains a `dashboard_url` like `/dashboard/<slug>`.
2. **Remove the legacy fallback.** Change line 808 to:
   ```js
   if (!result.dashboard_url) throw new Error('Signup succeeded but dashboard URL missing');
   dashboardLink.setAttribute('href', result.dashboard_url);
   ```
3. **Decide the fate of `dashboard.html`.** Either (a) redirect `/dashboard` → `/dashboard/<inferred-slug>` (requires a slug-resolver, probably via a cookie set at signup), or (b) replace `dashboard.html` with a small "Enter your merchant handle to log in" landing page that 302s to `/dashboard/<slug>`. Option (b) also fixes the nav link in `signup.html:262`.
4. **Update copy on the signup success screen.** Add a one-liner: "Your dashboard lives at **zettapay.vercel.app/dashboard/@your-handle** — bookmark it."

**Effort: M** — 1 API test + 1 client edit + 1 routing decision + 1 small landing page. Half-day for a single dev.

## Other gaps observed (lower priority — log only)

These were spotted but did not make the top 5. Listed for completeness; recommend triaging post-launch.

- **No `<link rel="canonical">` on any page.** Will cause duplicate-content signals if pages are reached via both `/pricing` and `/pricing.html`. Effort: S.
- **`checkout.html` missing `og:description`** specifically (`og:title` present). Subset of gap #1 but flagged separately because checkout is the most-shared URL type. Effort: S.
- **`contact.html:218` — `<select>` keyboard focus state is barely visible** (Tailwind default focus ring stripped). Effort: S.
- **Two coexisting dashboards** (`dashboard.html` legacy + `dashboard/index.html` new) — addressed indirectly by gap #5, but worth a dedicated mission to fully retire the legacy file. Effort: M.
- **Mobile viewport at 375px not empirically validated.** All pages set the viewport meta, but several heroes use `text-4xl`/`text-5xl` without `sm:` breakpoints, which can overflow on iPhone SE. Recommend a 30-min manual pass in DevTools device mode. Effort: S to audit, M-L to fix any breakage found.
- **Privacy + Terms generic.** Both files dated 2026-05-16 (today) but only mention third-party integrations (Coinflow, Circle, MoonPay, webhook) **4 times each combined**. Specifics on retention windows, EU/CCPA scope, and Solana protocol liability are thin. Effort: M (likely needs a legal review pass, not just engineering).
