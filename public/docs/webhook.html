Reference · webhooks

Webhook reference.

Stripe-grade webhooks for on-chain payments. HMAC-SHA256 signatures, 300 s tolerance window, at-least-once delivery with exponential backoff, idempotent by tx_hash.

Overview

When ZettaPay's chain listener detects a transaction that matches an open invoice — by destination address, amount, and (where supported) memo — it POSTs a signed JSON event to the HTTPS URL you registered at signup. The payload tells you which invoice was paid, on which network, with which tx_hash. You verify the signature, look up your order by customer_ref, mark it paid, and respond 2xx.

Delivery semantics

  • › Transport. HTTPS only. HTTP URLs are rejected at registration.
  • › Method. POST with Content-Type: application/json.
  • › Success. Any 2xx response within 10 s is treated as acknowledged.
  • › Failure. Non-2xx or timeout triggers retry with exponential backoff (see retries).
  • › Order. Events for the same invoice are delivered in confirmation order; cross-invoice ordering is not guaranteed.

Headers

HeaderValue
X-ZettaPay-Signature Hex-encoded HMAC-SHA256 of "{timestamp}.{raw_body}" using your webhook_secret.
X-ZettaPay-Timestamp Unix epoch seconds (UTC) when ZettaPay built the event. Reject if outside a 300 s tolerance window.
X-ZettaPay-Event-Id Stable id for this delivery attempt — useful for de-dupe and log correlation.
X-ZettaPay-Delivery-Attempt Integer, 1-based. First delivery is 1; retries increment.
User-Agent ZettaPay-Webhooks/1

Body schema

All events share the same envelope.

{
  "invoice_id":        "inv_01HMABCD...",
  "event":             "payment.confirmed",
  "amount":            "19.90",
  "currency":          "USDC",
  "chain":             "base",
  "tx_hash":           "5jkA...",
  "confirmations":     1,
  "customer_address":  "7Np41oeY...",
  "customer_ref":      "order_42",
  "created_at":        "2026-05-17T12:34:56Z"

invoice_id        string  · ZettaPay invoice id (inv_*)
event             string  · see "Event types" below
amount            string  · decimal as a string (avoid float drift)
currency          string  · "BTC" | "ETH" | "USDC"
chain             string  · "btc" | "base" | "polygon" | "ethereum"
tx_hash           string  · on-chain transaction hash / signature
confirmations     number  · confirmations at delivery time
customer_address  string  · the address that paid
customer_ref      string  · whatever you set on the invoice
created_at        string  · ISO-8601 UTC

Event types

EventFired when
payment.detected A matching transaction is observed in the mempool / unconfirmed. Optional — opt in per webhook endpoint.
payment.confirmed Transaction reached the configured confirmation threshold for the network. The signal you should act on.
payment.underpaid A confirmed transaction landed but was less than amount. Includes the actual amount received.
payment.overpaid Confirmed transaction exceeded amount (rare; useful for tipping flows).
invoice.expired Invoice hit its expiry window with no matching transaction. Safe to fail the order client-side.

Networks & finality

ZettaPay runs its own RPC listeners on every supported network. Default confirmation thresholds:

NetworkDefault confsTypical latency
basefinalized commitment~ 24 s
ethereum1~ 12 s
base1~ 2 s
polygon1~ 2 s

Reorgs. If a confirmed transaction is rolled back, ZettaPay re-fires the event when the new canonical chain re-includes (or omits) it. Always trust the latest event for a given tx_hash over a stale one.

Signature scheme

ZettaPay signs the concatenation "{X-ZettaPay-Timestamp}.{raw_body}" with HMAC-SHA256 using your webhook_secret, hex-encoded. To verify:

  1. Read the raw request body as bytes. Do not pre-parse JSON — re-serialization changes whitespace.
  2. Read X-ZettaPay-Signature and X-ZettaPay-Timestamp.
  3. Reject if |now - timestamp| > 300 seconds.
  4. Compute hmac_sha256(secret, f"{timestamp}.{body}"), hex-encode.
  5. Compare against the header using a constant-time comparator (crypto.timingSafeEqual, hmac.compare_digest, hash_equals, subtle::ConstantTimeEq).
  6. Process the event.

Verify examples

Pure-stdlib verification, no SDK required. Copy, drop into your handler, done. 

import { createHmac, timingSafeEqual } from 'node:crypto';

export function verifyZettaPay(
  rawBody: Buffer,
  signature: string,
  timestamp: string,
  secret: string,
  toleranceSec = 300,
): void {
  const ts = parseInt(timestamp, 10);
  if (!Number.isFinite(ts)) throw new Error('bad timestamp');

  const now = Math.floor(Date.now() / 1000);
  if (Math.abs(now - ts) > toleranceSec) {
    throw new Error('timestamp outside tolerance');
  }

  const expected = createHmac('sha256', secret)
    .update(`${timestamp}.${rawBody.toString('utf8')}`)
    .digest('hex');

  const sigBuf = Buffer.from(signature, 'hex');
  const expBuf = Buffer.from(expected, 'hex');
  if (sigBuf.length !== expBuf.length || !timingSafeEqual(sigBuf, expBuf)) {
    throw new Error('signature mismatch');
  }
}
import hmac
import hashlib
import time


class WebhookError(Exception): pass


def verify_zettapay(
    raw_body: bytes,
    signature: str,
    timestamp: str,
    secret: str,
    tolerance_sec: int = 300,
) -> None:
    try:
        ts = int(timestamp)
    except ValueError:
        raise WebhookError("bad timestamp")

    if abs(int(time.time()) - ts) > tolerance_sec:
        raise WebhookError("timestamp outside tolerance")

    payload  = f"{ts}.".encode() + raw_body
    expected = hmac.new(secret.encode(), payload, hashlib.sha256).hexdigest()

    if not hmac.compare_digest(expected, signature):
        raise WebhookError("signature mismatch")

declare(strict_types=1);

final class ZettaPayWebhook
{
    public static function verify(
        string $rawBody,
        string $signature,

use hmac::{Hmac, Mac};
use sha2::Sha256;
use std::time::{SystemTime, UNIX_EPOCH};
use subtle::ConstantTimeEq;

type HmacSha256 = Hmac<Sha256>;

pub fn verify_zettapay(
    raw_body: &[u8],
    signature_hex: &str,
    timestamp: &str,
    secret: &[u8],
    tolerance_sec: u64,
) -> Result<(), &'static str> {
    let ts: u64 = timestamp.parse().map_err(|_| "bad timestamp")?;
    let now  = SystemTime::now().duration_since(UNIX_EPOCH).unwrap().as_secs();
    if now.abs_diff(ts) > tolerance_sec {
        return Err("timestamp outside tolerance");
    }

    let mut mac = HmacSha256::new_from_slice(secret).map_err(|_| "bad key")?;
    mac.update(timestamp.as_bytes());
    mac.update(b".");
    mac.update(raw_body);
    let expected = mac.finalize().into_bytes();

    let sig = hex::decode(signature_hex).map_err(|_| "bad sig")?;
    if sig.ct_eq(&expected).into() { Ok(()) } else { Err("signature mismatch") }
}

Replay protection

The 300 s tolerance on X-ZettaPay-Timestamp bounds any attacker's replay window. Combine with idempotency for full coverage: even if an attacker captures a valid request within the window, your unique constraint on tx_hash ensures the order cannot be marked paid twice.

Clock drift. If your server's clock drifts > 5 minutes from UTC, signature verification will fail in ways that look like attacks. Run NTP. X-ZettaPay-Timestamp is always UTC seconds.

Idempotency

ZettaPay aims for exactly-once logical delivery: a successful event is never re-fired with the same tx_hash. In practice, network failures mean you may receive the same logical event more than once across retries — same tx_hash, same invoice_id. Defense in depth:

  • › Add a UNIQUE (tx_hash) constraint on your payments table; let the INSERT fail on duplicate.
  • › Or upsert by invoice_id and treat repeated calls as no-ops.
  • › Return 200 on duplicates so ZettaPay stops retrying.

Retry policy

Non-2xx responses (and 10 s timeouts) trigger retry with exponential backoff + full jitter:

After the final attempt, the event is parked in the dashboard's Webhook deliveries log and can be replayed manually. Successful events that you missed never go to a dead-letter queue silently — you can always replay from the log.

Testing locally

Expose your dev server with a tunnel and have ZettaPay deliver to it. 

# Expose your local Express / Flask / Slim server on a public HTTPS URL
ngrok http 3000

# Copy the https://*.ngrok.io URL into the merchant dashboard's
# Webhooks panel, append your route (e.g. /webhooks/zettapay).
# Same idea, with Cloudflare Tunnel
cloudflared tunnel --url http://localhost:3000
# Fire a synthetic payment.confirmed event at the URL you registered.
curl -X POST https://api.zettapay.dev/webhooks/test \
  -H "Authorization: Bearer $ZETTAPAY_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "event": "payment.confirmed", "chain": "base", "amount": "1.00" }'
